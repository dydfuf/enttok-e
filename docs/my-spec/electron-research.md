# Electron 추가 리서치: UI 라이브러리 도입 판단 & 추천

- **작성일**: 2025-12-19
- **기준 문서**: `docs/my-spec/research.md` (Electron + Python sidecar, Tiptap 기반 편집기, 로컬퍼스트 업무일지 앱)

---

## 결론

- **UI 라이브러리(컴포넌트/프리미티브) 도입은 유리합니다.**
  - 이 프로젝트는 “노트 편집 + 사이드 패널 + 검색 + 설정 + OAuth 연결 + 작업 진행률/로그 + 제안 카드”처럼 **상호작용 UI가 많고, 키보드/포커스/접근성 요구가 높아** 직접 구현 비용이 큽니다.
- **권장 1안(React 기준)**: **Radix UI Primitives + shadcn/ui(+ Tailwind)**
  - Radix: “동작/접근성/키보드 내비게이션/포커스 관리”를 갖춘 **헤드리스(unstyled) 프리미티브 레이어**
  - shadcn/ui: Radix 위에 얹는 **오픈 코드(copy-paste) 컴포넌트**로 빠르게 “우리 앱의 컴포넌트 라이브러리”를 구축
- **빠른 MVP가 최우선이면 대안**: **Mantine** 또는 **Radix Themes**
  - 단, 디자인/스타일링 제약과 장기 커스터마이징 비용(override/일관성 유지)을 고려해야 합니다.

---

## 1) 이 프로젝트에서 UI 라이브러리가 필요한 이유

`docs/my-spec/research.md`의 목표/구성요소를 UI 관점으로 풀면 아래 요구가 즉시 나옵니다.

- **리치 텍스트/마크다운 편집기 UX**
  - 에디터 자체(Tiptap) 외에도: 툴바, 링크/코드/이미지 삽입, slash-command, 포맷 팝오버, 멘션/태그 자동완성 등
- **데스크톱 앱다운 인터랙션**
  - 키보드 중심 탐색(커맨드 팔레트, 빠른 전환), 컨텍스트 메뉴, 토스트/알림, 다이얼로그(경고/확인)
- **설정/연결 UI**
  - OAuth 연결, 동기화 범위 선택(기간/채널/프로젝트), 권한/프라이버시 안내, 토큰/Keychain 연동, 로그/진행률 뷰
- **접근성(A11y)과 포커스 관리**
  - 모달/팝오버/드롭다운/메뉴는 “포커스 트랩, ESC 닫기, 탭 순환, 키보드 내비게이션”을 제대로 구현하지 않으면 품질이 크게 떨어집니다.

따라서 “버튼/인풋 같은 단순 위젯”을 넘어, **복잡한 UI 패턴(메뉴/다이얼로그/툴팁/토스트/탭/아코디언/스크롤 영역 등)**을 반복적으로 만들게 되므로 UI 라이브러리 도입이 개발 생산성과 품질에 직접적인 이득입니다.

## 2) Electron에서 UI 라이브러리를 고를 때 체크리스트

Electron 렌더러는 결국 웹 UI이지만, 앱 UX 요구가 더 빡센 편이라 아래 기준을 권장합니다.

- **접근성/키보드 내비게이션 품질**
  - WAI-ARIA 패턴 준수, focus management 제공 여부
- **커스터마이징 난이도**
  - “스타일이 강하게 고정된 라이브러리”는 처음은 빠르지만, 제품이 커질수록 override 비용이 누적됩니다.
- **테마(라이트/다크)와 디자인 토큰**
  - CSS variables 기반이면 Electron의 시스템 테마/고대비 모드 대응이 수월합니다.
- **데이터-덴스 UI 지원**
  - 좌/우 패널, 리사이즈, 트리/리스트/테이블, 긴 목록 성능(가상화) 등
- **라이선스/지속 가능성**
  - 상용 앱/배포를 고려하면 MIT 등 보수적인 라이선스가 마음 편합니다.

## 3) 후보 라이브러리 비교

아래는 “우리 앱에서 바로 쓸 수 있는가” 관점으로 정리했습니다.

### A. 헤드리스(unstyled) / 프리미티브 기반

#### 1) Radix UI Primitives

- **정의/성격**: 접근성과 커스터마이징에 집중한 **low-level UI 프리미티브(unstyled)**
- **강점(이 프로젝트에 특히 유리)**
  - WAI-ARIA 패턴을 따르며, **aria/role, focus management, keyboard navigation 같은 난이도 높은 부분을 처리**해줌
  - “필요한 것만 점진적으로 채택” 가능(Incremental adoption)
  - `radix-ui` 패키지 설치를 권장하며 tree-shakeable이라 **사용한 컴포넌트만 번들에 포함**될 수 있음
- **약점/주의**
  - 스타일은 직접 만들어야 하므로, “디자인 토큰/컴포넌트 스타일 규칙”을 프로젝트에서 정립해야 함
- **근거(공식 문서)**
  - [Radix Primitives 소개](https://www.radix-ui.com/primitives/docs/index)
  - [접근성/키보드/포커스 관리에 대한 설명](https://www.radix-ui.com/primitives/docs/overview/introduction)

#### 2) shadcn/ui

- **정의/성격**: 전통적인 “npm 설치형 컴포넌트 라이브러리”가 아니라, **컴포넌트를 프로젝트 코드로 가져와(오픈 코드) 우리 제품의 컴포넌트 라이브러리를 만드는 방식**
- **강점**
  - UI 컴포넌트 상단 레이어가 오픈 코드라 **스타일/동작을 제품 요구에 맞게 수정하기가 매우 쉬움**
  - 여러 라이브러리의 스타일 override 꼼수/랩핑 지옥을 줄이는 방향을 명시적으로 지향
  - CLI로 컴포넌트 추가 + 의존성 설치를 자동화
  - Radix + Tailwind 생태계 조합이 사실상 표준처럼 굳어져 있어, “커맨드 팔레트/메뉴/다이얼로그/토스트” 구성에 유리
- **약점/주의**
  - “가져온 코드”는 우리 코드베이스가 되므로 업데이트/버전업은 자동이 아니라 **선택적으로 반영**해야 함(장점이자 운영 부담)
- **근거(공식 문서)**
  - shadcn/ui는 “컴포넌트 라이브러리”라기보다 “컴포넌트 라이브러리를 만드는 방법”임을 명시: [shadcn/ui 소개](https://github.com/shadcn-ui/ui/blob/main/apps/v4/content/docs/(root)/index.mdx)

> 결론적으로, A그룹은 **장기 제품**에 유리합니다. “디자인은 우리만의 것”으로 가져가고, 접근성/동작은 검증된 프리미티브로 가져가는 구조입니다.

### B. 스타일 포함(완성형) 컴포넌트 라이브러리

#### 1) Mantine

- **정의/성격**: React 기반 “컴포넌트 + 훅”을 폭넓게 제공하는 올인원 UI 라이브러리
- **강점**
  - 템플릿/가이드가 잘 준비되어 있고 빠르게 화면을 만들기 쉬움
  - 라이선스가 MIT로 명시되어 상용 제품에서도 부담이 적음
- **약점/주의**
  - 제품 고유의 데스크톱 느낌(예: Obsidian/Notion류)으로 디자인을 강하게 커스터마이즈할수록, 라이브러리 기본 스타일/구조와 충돌하는 순간이 올 수 있음
- **근거(공식 문서)**
  - [Mantine Getting started](https://github.com/mantinedev/mantine/blob/master/apps/mantine.dev/src/pages/getting-started.mdx) (MIT 라이선스 명시 포함)

#### 2) Radix Themes

- **정의/성격**: Radix Primitives 위에, 접근성과 일관된 스타일/테마를 얹은 “프리-스타일드(pre-styled)” 컴포넌트 세트
- **강점**
  - 라이트/다크, accent color, radius, scale 등 **테마를 전역으로 바꾸는 기능**을 제공
  - “아름다운 기본값 + 접근성 기본 내장”을 강하게 지향
- **약점/주의**
  - 제품 UI를 아주 독특하게 만들려면(예: 완전한 네이티브 느낌/브랜드 강함) 결국 커스텀 레이어가 많이 필요할 수 있음
- **근거(공식 문서/요약)**
  - [Radix Themes 리포지토리](https://github.com/radix-ui/themes)

#### 3) MUI(Material UI)

- **정의/성격**: Material Design 기반의 대중적인 React 컴포넌트 라이브러리
- **장점**
  - 컴포넌트 수/레퍼런스가 매우 많고, 팀 경험자도 많음
- **단점(이 프로젝트 기준)**
  - Material 디자인이 “데스크톱 노트 앱”의 톤과 맞지 않으면, 커스터마이징 비용이 커질 수 있음

#### 4) Fluent UI

- **정의/성격**: Microsoft의 Fluent Design 기반 React 컴포넌트/유틸리티
- **장점**
  - Microsoft 제품군에서 검증된 컴포넌트
  - v8/v9 공존 및 점진 마이그레이션을 지원한다고 명시
- **단점(이 프로젝트 기준)**
  - 윈도우/오피스 톤이 강해, macOS 사용자에게는 “웹앱 느낌/이질감”이 생길 수 있음
- **근거(공식 문서)**
  - [Fluent UI 저장소 README](https://github.com/microsoft/fluentui)

#### 5) Blueprint.js

- **정의/성격**: Palantir의 React 기반 UI 툴킷(데이터-덴스 UI에 강점)
- **장점**
  - 테이블/리스트/필터 같은 “업무용 UI” 톤에 적합한 편
- **단점/주의**
  - CSS/리소스 import 구조가 뚜렷해서 초기 세팅을 정확히 해야 함
- **근거(공식 문서)**
  - [Blueprint Getting started](https://github.com/palantir/blueprint/blob/develop/packages/docs-app/src/getting-started.md)

## 4) 추천안 상세

### 권장 1안: Radix UI Primitives + shadcn/ui (+ Tailwind)

이 조합을 “권장 1안”으로 두는 이유는 다음과 같습니다.

- **제품이 커질수록 이득**: 데스크톱 앱은 “설정/팝오버/메뉴/단축키/포커스” 같은 난이도 높은 UI가 계속 추가됩니다.
  - Radix는 이러한 **접근성/키보드/포커스 관리**를 라이브러리 레벨에서 처리하는 것을 목표로 합니다.
- **디자인 자유도 최고**: shadcn/ui는 “오픈 코드”를 원칙으로 하여, 디자인 시스템에 맞춰 수정/확장하기가 쉽습니다.
- **AI 기반 개발(이 프로젝트 방향)과 궁합**: 컴포넌트 코드가 프로젝트에 존재하므로, LLM이 컴포넌트를 직접 읽고 개선/추가 생성하기가 쉽습니다(문서에서도 AI-Ready를 강조).

#### 도입 전략(현실적인 운영 방식)

- **초기(MVP)**
  - shadcn/ui로 기본 세트부터: `Button`, `Input`, `Dialog`, `DropdownMenu`, `Tabs`, `Tooltip`, `Toast`, `Command`(커맨드 팔레트), `Form` 류
  - “앱 레이아웃(좌/우 패널 + 에디터)”는 별도 레이아웃 컴포넌트로 구성(리사이즈 패널은 서드파티를 붙여도 됨)
- **확장**
  - 제품 요구에 따라 shadcn/ui 컴포넌트를 수정해 “우리 앱 스타일”로 굳히고, 부족한 패턴은 Radix primitive를 직접 조합해서 추가

### 대안 1: Mantine (속도 최우선)

- 화면을 빠르게 찍어내야 하고, 디자인 커스터마이징 요구가 상대적으로 낮다면 Mantine이 MVP에 유리할 수 있습니다.
- 다만 “Obsidian/Notion 같은 데스크톱다운 톤”을 강하게 원하면, 장기적으로는 권장 1안(헤드리스 기반)이 더 안정적입니다.

### 대안 2: Radix Themes (빠른 프로토타이핑)

- “Radix 철학(접근성) + 프리-스타일드 컴포넌트”의 중간 지점입니다.
- 디자인 자유도가 완전 헤드리스보다 줄어드는 대신, 초반 완성도를 빠르게 올릴 수 있습니다.

## 5) Electron UI 구현 시 추가 주의점(라이브러리와 무관하지만 영향 큼)

- **커스텀 타이틀바/드래그 영역**: 커스텀 헤더를 만들면 `-webkit-app-region: drag` / `no-drag` 처리가 필요합니다.
  - UI 라이브러리 버튼/입력 요소가 드래그 영역에 들어가면 클릭이 씹힐 수 있으므로, 헤더 영역 설계 시 미리 규칙화해야 합니다.
- **포커스/단축키 충돌**: 에디터(Tiptap) 내부 단축키와 앱 전역 단축키(커맨드 팔레트 등)가 충돌하기 쉬워, “전역 핫키는 에디터 focus 상태를 고려”하는 설계가 필요합니다.
